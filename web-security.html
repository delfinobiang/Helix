HTTPS & TLS

HTTPS is HTTP run over TLS (Transport Layer Security). It provides three guarantees: confidentiality (data is encrypted), integrity (data can't be tampered with), and authentication (you're talking to who you think you are).

🔐 TLS Handshake Overview
  • 1. Client Hello — browser sends supported TLS versions, cipher suites, a random nonce
  • 2. Server Hello — server picks a cipher, sends its certificate
  • 3. Certificate Validation — browser verifies the cert chain to a trusted CA
  • 4. Key Exchange — both sides derive a shared session key (without transmitting it)
  • 5. Encrypted Traffic — all further data is encrypted with the session key
📜 Certificates & CAs

A TLS certificate proves the server is who it claims to be. It's issued by a Certificate Authority (CA) that your OS and browser inherently trust. Services like Let's Encrypt provide free certificates. Without a valid cert, browsers show a security warning.

# Check a site's certificate openssl s_client -connect example.com:443 -showcerts # HTTP headers that improve security Strict-Transport-Security: max-age=31536000; includeSubDomains # Forces HTTPS for 1 year — even if user types http:// Content-Security-Policy: default-src 'self' # Restricts what resources the page can load X-Frame-Options: DENY # Prevents clickjacking by disallowing iframes

Lesson 02 · Web & Internet

Authentication

Authentication answers "who are you?" — verifying identity. Authorisation answers "what can you do?" — controlling access. They are distinct problems and must be solved separately.

🎟️ Session vs Token Auth
  • Session cookies — server stores session in DB; client holds a session ID in a cookie. Stateful. Easier to revoke.
  • JWT (JSON Web Token) — self-contained token signed by the server. Stateless. Client sends it in every request header. Cannot easily be revoked.
  • OAuth 2.0 — delegated authorisation (Sign in with Google/GitHub). You don't handle passwords.

✅ Password Best Practices
  • Never store passwords in plaintext — always hash with bcrypt, Argon2, or scrypt
  • Add a unique salt per password to defeat rainbow tables
  • Use bcrypt with cost factor ≥ 12 for new systems
  • Implement rate limiting and lockout to prevent brute force
  • Offer MFA — it's the single best security upgrade available

Secure Coding

Security isn't a feature you add at the end — it's a discipline baked into every line of code. Following secure coding principles dramatically reduces your attack surface.

🛡️ Core Security Principles
  • Least Privilege — grant only the minimum permissions needed; a DB user for read queries shouldn't have DELETE access
  • Defence in Depth — multiple independent security layers; one failure shouldn't compromise everything
  • Never Trust User Input — validate, sanitise, and escape all input on the server
  • Fail Secure — when something goes wrong, default to the more secure state (deny, not allow)
  • Keep Secrets Secret — never commit API keys, passwords, or tokens to version control
# Environment variables for secrets import os DB_PASSWORD = os.environ['DB_PASSWORD'] # ✓ DB_PASSWORD = "hunter2" # ✗ NEVER # Input validation def get_user(user_id): if not isinstance(user_id, int) or user_id < 1: raise ValueError("Invalid user ID") return db.query("SELECT * FROM users WHERE id=?", (user_id,)) # Rate limiting (pseudo-code) if login_attempts(ip) > 5: raise TooManyRequests("Try again in 15 minutes")
🔒 Security Headers Checklist
  • HSTS — enforces HTTPS for future visits
  • CSP — controls what resources can be loaded (XSS mitigation)
  • X-Frame-Options — prevents clickjacking
  • X-Content-Type-Options: nosniff — prevents MIME sniffing attacks
  • Referrer-Policy — controls what's in the Referer header
