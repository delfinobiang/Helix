Lesson 01 · Web & Internet

What is the Internet?

The internet is a global network of networks — billions of devices connected by physical cables, fibre optics, and wireless links. Understanding its structure reveals why it's so resilient and why it works the way it does.

🌐 The Big Picture

The internet isn't one thing — it's a collection of independently-operated networks (called Autonomous Systems) that agree to exchange data using common protocols. No single entity owns it. It was designed to survive partial outages by routing around damage.

🏗️ Key Infrastructure Layers
  • Tier 1 ISPs — global backbone networks connected by undersea cables
  • Tier 2 ISPs — regional providers who buy transit from Tier 1
  • Tier 3 ISPs — local providers who sell broadband to end users
  • CDNs — Content Delivery Networks that cache content close to users

When you load a webpage, your request might travel through dozens of different networks and physical locations — yet it all appears seamless. That abstraction is the internet's greatest achievement.

Lesson 02 · Web & Internet

DNS & IP Addresses

Every device on the internet has an IP address — a numerical label that identifies it. But humans prefer names like google.com. The Domain Name System (DNS) is the internet's phonebook, translating names into numbers.

📖 DNS Resolution Steps
  • 1. Browser cache — check if the IP was looked up recently
  • 2. OS cache / hosts file — check local system records
  • 3. Recursive resolver — your ISP's DNS server
  • 4. Root nameserver — directs to the TLD server (.com, .org)
  • 5. TLD nameserver — directs to the authoritative nameserver
  • 6. Authoritative nameserver — returns the final IP address
# IPv4 — 32-bit, ~4 billion addresses 142.250.80.14 # google.com (example) # IPv6 — 128-bit, virtually unlimited 2607:f8b0:4004:c08::71 # google.com IPv6 # DNS record types A → maps domain to IPv4 address AAAA → maps domain to IPv6 address CNAME → alias pointing to another domain MX → mail server for the domain TXT → arbitrary text (used for verification)
⚡ TTL & Caching

Each DNS record has a Time To Live (TTL) — how many seconds it can be cached before re-querying. A TTL of 3600 means the IP is cached for an hour. This is why DNS changes can take time to propagate globally.

TCP/IP & Packets

Data on the internet travels as packets — small chunks that may take different routes and arrive out of order. The TCP/IP protocol suite handles addressing, routing, and reassembly to give you a seamless experience.

📦 How Packets Work

A large file is split into thousands of packets. Each packet carries: a header (source IP, destination IP, sequence number, checksum) and a payload (chunk of data). Routers forward each packet independently toward the destination, where TCP reassembles them in the correct order.

# TCP Three-Way Handshake (connection setup) Client → Server: SYN # "I want to connect" Server → Client: SYN-ACK # "OK, I'm ready" Client → Server: ACK # "Great, let's go" # Connection is now established — data flows # TCP vs UDP TCP: reliable, ordered, error-checked → web, email, files UDP: fast, no guarantees → video calls, games, DNS
🔢 The OSI Model
  • 7 — Application: HTTP, FTP, SMTP
  • 4 — Transport: TCP, UDP (segments)
  • 3 — Network: IP (packets)
  • 2 — Data Link: Ethernet, Wi-Fi (frames)
  • 1 — Physical: bits on the wire

Lesson 04 · Web & Internet

The Request-Response Cycle

Every time you load a page, your browser and a server have a structured conversation. Understanding this cycle is fundamental to everything in web development — from debugging to performance optimisation.

Use the simulator below to walk through each step of a real browser request:

🌐 Request Flow Simulator

🖥️
1. Browser — User types URL
You type https://example.com and press Enter. The browser begins the process.
🔍
2. DNS Lookup
Browser asks DNS: "What IP is example.com?" DNS responds: 93.184.216.34
🤝
3. TCP + TLS Handshake
Browser opens a TCP connection on port 443. TLS negotiates encryption keys — the connection is now secure.
📤
4. HTTP Request Sent
Browser sends: GET / HTTP/2 with headers (Host, User-Agent, Accept, etc.)
⚙️
5. Server Processing
Server receives request, processes it (database query, rendering, etc.) and prepares a response.
📥
6. HTTP Response
Server replies: 200 OK + HTML body. Browser begins rendering as bytes arrive.
🎨
7. Browser Renders Page
Browser parses HTML, fetches CSS/JS/images (more requests!), constructs the DOM, and paints pixels to your screen.
✅ Key Takeaways
  • A single page load can trigger dozens of HTTP requests (HTML, CSS, JS, images, fonts)
  • DNS resolution adds latency — browsers cache IPs to speed up repeat visits
  • TLS handshake adds ~1 RTT — HTTP/2 and HTTP/3 reduce this overhead significantly
  • The browser renders progressively — content appears before all resources finish loading
