Lesson 01 · Web & Internet

HTTP Methods

HTTP (HyperText Transfer Protocol) is the language browsers and servers speak. Every request has a method — a verb that tells the server what action to perform.

📋 The Main HTTP Methods
  • GET — retrieve a resource; no body; safe & idempotent
  • POST — create a new resource; includes a body; not idempotent
  • PUT — replace a resource entirely; idempotent
  • PATCH — partially update a resource
  • DELETE — remove a resource; idempotent
  • HEAD — like GET but returns headers only (useful for checking if a resource exists)
  • OPTIONS — ask the server what methods it supports (used in CORS)
# GET — fetch a user GET /api/users/42 HTTP/1.1 Host: api.example.com Authorization: Bearer eyJhbGci... # POST — create a new user POST /api/users HTTP/1.1 Content-Type: application/json {"name": "Alice", "email": "alice@example.com"} # DELETE — remove a user DELETE /api/users/42 HTTP/1.1
🔄 Idempotency

A request is idempotent if calling it multiple times has the same effect as calling it once. GET, PUT, DELETE are idempotent. POST is not — sending the same POST twice might create two records.

Challenge 1+15 XP

Create vs Retrieve

You're building a sign-up form. When the user submits their details to create a new account, which HTTP method should the request use? Type: POST

Creating a new resource = POST. Retrieving = GET.

Status Codes

Every HTTP response includes a status code — a three-digit number that tells the client what happened. The first digit indicates the class of response.

RangeCategoryMeaning
1xxInformationalRequest received, processing
2xxSuccessRequest succeeded
3xxRedirectionFurther action needed
4xxClient ErrorBad request (your fault)
5xxServer ErrorServer failed (their fault)
🔢 Must-Know Codes

200 OK Standard success

201 Created Resource created (POST)

301 Moved Permanently Redirect

400 Bad Request Malformed request

401 Unauthorized Not authenticated

403 Forbidden Authenticated but no permission

404 Not Found Resource doesn't exist

500 Internal Server Error Generic server crash

503 Service Unavailable Server down/overloaded

Lesson 03 · Web & Internet

REST APIs

A REST API (Representational State Transfer) is a set of conventions for building web APIs using HTTP. It's not a protocol — it's an architectural style. Well-designed REST APIs are predictable, stateless, and resource-oriented.

📐 REST Constraints
  • Stateless — each request contains all info needed; no server-side session
  • Resource-based — URLs identify resources, not actions (/users/42 not /getUser?id=42)
  • Uniform interface — HTTP methods define actions on resources
  • Client-server — frontend and backend are independent
# RESTful URL design GET /posts # list all posts POST /posts # create a new post GET /posts/7 # get post #7 PUT /posts/7 # replace post #7 PATCH /posts/7 # update post #7 fields DELETE /posts/7 # delete post #7 # Nested resources GET /posts/7/comments # comments on post 7 POST /posts/7/comments # add comment to post 7
🆚 REST vs GraphQL vs gRPC

REST — simple, widely understood, great for public APIs. GraphQL — client specifies exactly what data it needs, avoids over/under-fetching. gRPC — binary protocol, extremely fast, great for internal microservices.

Challenge 3+15 XP

Statelessness

REST APIs are stateless. This means the server stores no session state between requests — each request must contain all needed information. What must every authenticated request include? Type: token

Since there's no server session, identity must travel with the request — usually as a JWT or API key.
Lesson 04 · Web & Internet

Fetch & JSON

JSON (JavaScript Object Notation) is the lingua franca of web APIs. The browser's built-in Fetch API lets you make HTTP requests from JavaScript and handle responses asynchronously.

// GET request with fetch const response = await fetch('https://api.example.com/users/1'); const user = await response.json(); console.log(user.name); // "Alice" // POST request with JSON body const res = await fetch('/api/posts', { method: 'POST', headers: { 'Content-Type': 'application/json' }, body: JSON.stringify({ title: 'Hello World', body: '...' }) }); const created = await res.json(); console.log(res.status); // 201 // Always handle errors if (!res.ok) throw new Error(`HTTP error: ${res.status}`);

🛡️ CORS

Cross-Origin Resource Sharing — browsers block JS from fetching resources on different domains by default. The server must include Access-Control-Allow-Origin headers to permit cross-origin requests. This is why your frontend dev server needs a proxy when calling a different-origin API.
