Why Efficiency Matters

Two programs that produce the same output can differ by millions of times in speed. Understanding algorithmic efficiency is what separates code that works from code that scales.

⚡ The Real Cost of Slow Code

A loop that runs n² times works fine for n=100 (10,000 steps). For n=1,000,000 it runs one trillion steps. No amount of fast hardware fixes bad algorithms at scale.

📈 Growth Rates Ranked (best → worst)
  • O(1) — Constant: same time regardless of input
  • O(log n) — Logarithmic: binary search, tree ops
  • O(n) — Linear: single loop over all elements
  • O(n log n) — Linearithmic: efficient sorting
  • O(n²) — Quadratic: nested loops
  • O(2ⁿ) — Exponential: avoid at all costs

Big O Notation

Big O notation describes the upper bound on how a function grows as its input grows. We drop constants and lower-order terms — only the dominant growth pattern matters.

📐 The Rule of Simplification

O(3n² + 5n + 100) simplifies to O(n²). We keep only the fastest-growing term and drop constants. At large n, the n² term dominates everything else completely.

# O(1) — constant time def first_element(arr): return arr[0] # always 1 step # O(n) — linear time def find_max(arr): for x in arr: # n steps if x > best: best = x return best # O(n²) — quadratic time def bubble_sort(arr): for i in range(len(arr)): # n for j in range(len(arr)): # n → total n² if arr[j] > arr[j+1]: ... # O(log n) — binary search def binary_search(arr, target): lo, hi = 0, len(arr)-1 while lo <= hi: mid = (lo+hi)//2 if arr[mid] == target: return mid elif arr[mid] < target: lo = mid+1 else: hi = mid-1 # halves each step

Space Complexity

Big O applies to memory usage too, not just time. Space complexity measures how much extra memory an algorithm uses relative to its input size.

💾 Time vs Space Trade-offs

Faster algorithms often use more memory. Caching (memoization) trades space for speed. Streaming algorithms trade speed for minimal memory. Understanding both lets you choose the right tool.

Lesson 04 · CS Theory

Analysing Code

Reading code and determining its Big O is a core skill. There are simple rules that let you analyse almost any algorithm systematically.

📋 The Rules of Analysis
  • Single loop: O(n) — runs once per element
  • Nested loops: O(n²) — outer × inner steps
  • Halving each step: O(log n) — binary search, tree traversal
  • Loop + halving: O(n log n) — merge sort, heap sort
  • Sequential (not nested) loops: O(n) — add, then pick dominant
# Analyse this: what's the Big O? def mystery(arr): result = [] # O(1) for x in arr: # O(n) for y in arr: # O(n) nested result.append(x * y) # → O(n²) total return result # What about this? def two_loops(arr): s = 0 for x in arr: s += x # O(n) for x in arr: s -= x # O(n) return s # O(n) + O(n) = O(2n) = O(n)

📦
Databases
B-tree indexes make O(log n) queries possible — finding one row in a billion-row table in microseconds.
